# 大福的agent学习101｜从 LLM 调用、单 Agent 到多 Agent 编排，一步步跑通

这是「大福的agent学习101」第一部分。

这一系列我会继续往后写，后面会逐步补上更多关于 RAG、LangChain、LangGraph、检索优化和工程化实践的学习内容。

如果你最近也在学 Agent，但总觉得概念很多、工程很散、看完文章还是不知道该从哪下手，这个项目就是按“能看懂、能跑通、能自己改”的思路做的。

这个仓库对应的价值，不只是放几个 demo，而是把我在 `docs/` 里整理的学习笔记，尽量落成了可以运行的代码。

它不是一个追求炫技的复杂框架，而是一个偏**学习型、拆解型、演示型**的开源项目：

- 先从最基础的大模型 SDK 调用开始
- 再过渡到一个可运行的单 Agent
- 再继续到多 Agent 的任务拆解与汇总
- 最后补上 Embedding、RAG、Rerank、LangGraph 这些常见能力

项目地址：`https://github.com/KaiEi1020/my_agent_project`

也可以在掘金里直接放成超链接：[GitHub - my_agent_project](https://github.com/KaiEi1020/my_agent_project)

这个系列后续的代码和文章更新，也都会继续落在这个仓库里。

## 先说结论：这个项目适合谁

如果你属于下面几类人，这个项目会比较合适：

| 人群 | 你会得到什么 |
|---|---|
| 刚开始学 Agent 的同学 | 能从 0 跑通一个最小可用 Agent |

| 想理解多 Agent 编排的同学 | 能看到 master-worker 模式最小实现 |

| 不想一上来就进大框架黑盒的人 | 能先理解底层，再看 LangChain / LangGraph |

这也是我做这个仓库的初衷：**把 Agent 学习路径拆细，而不是一上来把所有抽象都堆上去。**

## 这个项目里有什么

当前仓库大致分成 4 层：

| 层级 | 路径 | 作用 |
|---|---|---|
| LLM 基础层 | `src/llm/` | 封装模型调用、system prompt 注入、结构化输出校验 |
| Prompt 工具层 | `src/prompts/` | 统一清理多行 prompt 缩进 |
| Agent 层 | `src/my_agent/` | 单 Agent 与多 Agent 编排逻辑 |
| Demo 层 | `demos/` | 各类可直接运行的学习示例 |

从学习顺序上，我更建议把它理解成一条逐步递进的路线：

```text
SDK 调用 -> 结构化输出 -> 单 Agent -> 多 Agent -> Embedding/RAG -> 重排 -> LangGraph
```

这条路线的好处是：你不会只停留在“会调用某个框架 API”，而是能知道每一层到底解决了什么问题。

## 这个项目最有价值的地方，不是“能跑”，而是“能拆”

很多 Agent 项目最大的问题不是做不出来，而是**新人读不进去**。

要么上来就是一大坨框架封装；
要么是业务代码混在一起；
要么是效果看起来很炫，但不知道核心机制在哪里。

这个项目我比较看重三点：

| 特点 | 说明 |
|---|---|
| 结构清晰 | 底层能力、prompt、agent、demo 是分层的 |
| 路径渐进 | 从简单调用到多 Agent，再到 RAG 能力，学习链路比较顺 |
| 可改可观察 | 很多 demo 都是“能直接跑、能打印过程、能动手改”的 |

尤其是当你学习 Agent 时，**“看到中间过程”非常重要**。因为 Agent 不是只看最终答案，更关键的是：

- 它为什么要调用这个工具
- 它工具参数是怎么组织的
- 工具返回后它如何继续下一轮推理
- 它什么时候决定输出 final answer

这些，在这个项目里都能看到比较清楚的最小实现。

## 核心一：先把 LLM 基础调用封装干净

项目里的 `BaseLLMCaller` 负责比较基础但非常关键的几件事：

| 能力 | 作用 |
|---|---|
| 客户端初始化 | 统一初始化智谱 `ZhipuAiClient` |
| 消息构造 | 自动注入基础 `system_prompt` |
| JSON 提取 | 兼容模型返回代码块 JSON 或纯 JSON |
| 结构化输出 | 把 Pydantic Schema 转成 JSON Schema 注入提示词，并做本地校验 |

这部分很重要，因为很多人一开始写 Agent，最大的问题不是“不会写循环”，而是**输出不稳**。

模型一旦不稳定输出，你后面的工具调用、状态流转、结果汇总都会跟着崩。

所以这里做了一件很实用的事：把结构化输出收口到统一方法里，而不是在业务代码里到处手搓 JSON 解析。

这个思路其实很适合学习期：

1. 先让模型输出变稳定
2. 再让工具调用变稳定
3. 最后再去扩展更复杂的 Agent 行为

这比一开始就追求“智能感”更重要。

## 核心二：用一个最小 MiniAgent 跑通 ReAct

项目里的 `MiniAgent` 是我觉得很适合拿来学习的部分。

它本质上是一个简化版 ReAct Agent，核心链路很直接：

1. 根据角色 prompt 和工具描述，组装完整 system prompt
2. 调用模型，要求它输出结构化 JSON
3. 把输出解析成两类结果：`tool_call` 或 `final_answer`
4. 如果是工具调用，就执行工具
5. 把工具结果作为 `Observation` 回灌给模型，继续下一轮

这其实就是很多 Agent 的骨架。

### 为什么这个实现值得看

因为它不是只给你一个“Agent 对象”，而是把几个关键细节都摊开了：

| 细节 | 在学习中为什么重要 |
|---|---|
| 工具描述来自函数 docstring | 你能直观看到 docstring 会怎样影响模型理解工具 |
| 工具参数来自函数签名 | 你会知道为什么参数名设计很重要 |
| JSON 结构有明确约束 | 你能理解为什么 Agent 输出要尽量收敛 |
| Observation 回灌到消息历史 | 你能看清工具执行与下一轮推理的连接点 |

很多人学 Agent 时会忽略一个点：**工具不是“挂上去就行”，工具的 docstring 和签名本身就是 prompt 的一部分。**

这个项目把这件事展示得很直接。

## 核心三：多 Agent 不是玄学，本质上就是拆任务、分发、汇总

项目里的 `MultiAgentOrchestrator` 走的是很经典的 master-worker 模式。

它做的事情可以概括成三步：

| 步骤 | 说明 |
|---|---|
| 拆解 | 先把用户请求拆成多个 `SubTask` |
| 分发 | 按 `worker_type` 把子任务交给对应专家 Agent |
| 汇总 | 收集各专家结果，再生成统一答复 |

这类设计非常适合用来理解“多 Agent 到底比单 Agent 多了什么”。

答案其实很简单：

**多 Agent 不是更神秘，而是把单个复杂问题拆成多个相对聚焦的问题。**

例如这个项目里的 demo，会把一个复合任务拆给：

- 售后专家 Agent
- 行程 / 机票专家 Agent

然后再由调度器统一汇总结果。

这对初学者特别友好，因为你能立刻理解：

- 为什么要拆任务
- 什么情况下需要不同 worker
- 为什么最终还需要一个“总经理”做总结

相比一开始就上很复杂的 graph 或 runtime，这种 master-worker 的学习成本低很多，但认知价值很高。

## 怎么跑这个项目

如果你准备把这篇文章发到掘金，这一段最好写得非常直接，因为读者最在意“我能不能 5 分钟跑起来”。

### 第一步：先准备智谱账号

先去智谱 GLM 注册一个新账号，领取新人礼包。

### 第二步：配置环境变量

在项目根目录下，参考 `.env-example` 填写环境变量：

```bash
ZAI_API_KEY=
ZAI_MODEL=
```

### 第三步：安装依赖

```bash
uv sync
```

### 第四步：运行 demo

```bash
uv run demos/xxx.py
```

如果你想按学习路径来跑，可以直接这样试：

| 目标 | 命令 |

| 跑基础模型调用 | `uv run demos/demo_01__glm_basic.py` |
| 跑单 Agent 售后 demo | `uv run demos/demo_02_after_sales_agent.py` |

## 这个项目适合怎么学

我不建议把这个仓库当成“拿来即用的生产框架”，更建议把它当成一个**Agent 学习沙盘**。

我的建议学习方式是：

### 1. 不要只跑通，要边跑边改

比如你可以改这些地方：

| 你可以改什么 | 你会观察到什么 |
|---|---|
| 改 system prompt | Agent 的角色行为会变化 |
| 改工具 docstring | 模型对工具用途的理解会变化 |
| 改函数签名 | 工具参数生成方式会变化 |
| 改结构化输出 schema | 模型输出约束会变化 |
| 改 max_steps | Agent 的探索轮次会变化 |

### 2. 多看打印过程，不要只看最终结果

尤其是 `MiniAgent` 这种实现，最值得看的不是最后那句结果，而是中间这些过程：

- LLM 返回了什么 JSON
- 解析后的 payload 是什么
- 调用了哪个工具
- Observation 是什么
- 为什么继续下一轮
- 为什么结束

你把这些看懂了，后面无论换到别的框架，理解都会快很多。

### 3. 先理解“为什么这样设计”，再去套框架

很多时候不是框架难，而是底层概念没捋清。

比如：

- 为什么多行 prompt 要统一清理缩进
- 为什么结构化输出要统一收口
- 为什么工具描述要从 docstring 里提取
- 为什么多 Agent 需要一个调度器
- 为什么 RAG 检索后还要重排

这些问题搞懂以后，你再去看更复杂的 Agent 框架，会轻松很多。

## 这个项目给我的一个很深的感受

过去大家学大模型应用，很多时候是“会调一个 API”。

但到了 Agent 阶段，真正的门槛已经变成了：

| 能力 | 为什么重要 |
|---|---|
| Prompt 组织能力 | 决定模型行为边界 |
| 结构化输出约束能力 | 决定系统稳不稳定 |
| 工具抽象能力 | 决定 Agent 能不能真正行动 |
| 编排能力 | 决定复杂任务能否拆开处理 |
| 检索与重排能力 | 决定知识型任务答得准不准 |

所以我越来越觉得，学习 Agent 最好的方式不是刷概念，而是做这种**能把每一层拆开的最小项目**。

这个仓库本质上就是在做这件事：

**把“Agent 是什么”从一个抽象词，变成一组可以逐步运行、逐步修改、逐步理解的代码。**

## 最后

如果你正准备系统学 Agent，我会很建议你先从这种“能直接看到骨架”的项目开始。

因为很多时候，真正阻碍学习的不是代码量，而是抽象层太高。

这个开源项目的价值，就在于它把 Agent 学习过程中那些最关键、最容易模糊的点，都尽量摊平了：

- 大模型怎么调
- 结构化输出怎么约束
- 单 Agent 怎么形成闭环
- 多 Agent 怎么拆解协作
- RAG 链路怎么逐步补齐

如果你想学的不是“某个框架怎么用”，而是**Agent 为什么这样工作**，那这个项目会是一个不错的起点。
