# 多源数据知识图谱 (GraphRAG)

## 1. RAG 短板

> **最根本的矛盾：检索是离散匹配，推理是连续理解。**

传统 RAG 把文档切成碎片，用向量相似度做"点对点"匹配。但用户真正想问的，往往是需要串联、归纳、跨越多个实体的连续推理。离散匹配与连续理解之间的鸿沟，导致了以下三重短板：

### 1.1 多跳推理（Multi-hop）无力

```tex
Q: "A 公司的 CEO 毕业于哪所大学？"
→ 需要先查 CEO 是谁，再查其教育背景
→ 传统 RAG 一次检索搞不定
```

---

## 2. GraphRAG 核心思想

> **检索的不是文本片段，而是实体、关系和推理路径。**

将文本转化为知识图谱（节点=实体，边=关系），使 LLM 能够：
- **多跳推理** — 沿关系链逐层跳转
- **全局聚合** — 基于社区摘要回答趋势性问题
- **隐性关联** — 发现"供应商的供应商"等间接关系

---

## 3. 工作流

```
文档输入 → 文本分块 → 实体/关系抽取 → 构建知识图谱 → 社区检测/摘要 → 子图检索 → 答案生成
```

| 步骤 | 技术要点 |
|------|----------|
| 文本分块 | 重叠窗口、语义边界感知 |
| 实体/关系抽取 | LLM + Pydantic 结构化输出，预定义 Schema |
| 图谱构建 | Neo4j / 内存图 |
| 社区检测 | Leiden 算法聚类，生成社区摘要 |
| 查询路由 | 局部查询(实体匹配) vs 全局查询(社区摘要) |
| 子图检索 | 多跳遍历、路径排序 |

---

## 4. 与传统 RAG 对比

| 维度 | 传统 RAG | GraphRAG |
|------|----------|----------|
| 检索单元 | 文本 Chunk | 实体 + 关系 + 社区摘要 |
| 推理能力 | 单跳（相似度） | 多跳（图遍历） |
| 全局问题 | 无法回答 | 社区摘要聚合 |
| 可解释性 | 黑盒相似度 | 白盒路径溯源 |
| 适用场景 | 事实问答 | 分析推理、关联挖掘 |
