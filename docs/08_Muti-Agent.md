# Multi-Agent 

## 为什么需要多智能体

**角色专业化**：单一 LLM 往往难以同时精通代码编写、需求分析和测试反馈。通过拆分任务，让专精的 Agent 各司其职，能显著提升交付质量。

**错误纠偏机制**：通过“规划者-执行者-审查者”的闭环，Agent 之间可以相互校验

**复杂任务拆解**：针对长逻辑链条任务，协作模式可以将大目标拆分为子任务，动态分配给不同的智能体单元



## 编排体系（思维图 GoT 的终极形态）。
### 1. 流水线模式 (Sequential)
Agent A 处理完传给 Agent B，单向流动。

###  2. 主管集中路由架构 (Supervisor)    /  星型模式 (Hub-and-Spoke)

**Supervisor（主管）：** 它就是那个 **Hub**。通常由一个强大的 LLM 担任，扮演路由器的角色。它理解整体目标，决定下一步哪个子 Agent（Worker）该干活。

**Workers（员工）：** 它们就是 **Spokes**。各自拥有特定的 Tool（如 Search、CodeExecutor），干完活后把状态和结果返回给 Supervisor。



一个中心 Agent（主管）负责分发任务，其他 Agent（下属）负责执行，最后汇总
  - 缺点：中心主观的llm会成为整个系统的“性能瓶颈”和“单点故障源”
  -  防止中心 Hub 崩溃
    1. 显式计数器与硬限: 必须为 Hub 的决策循环设置一个硬上限（比如 `max_turns = 10`）。一旦超过这个次数，系统必须强制中断，并触发人工介入（Human-in-the-Loop）或返回友好的错误提示
    2. 状态裁剪与“记忆擦除”（State Truncation）：
       1. 专门有一个轻量级的总结机制，把之前的讨论历史压缩成一份当前的 `State Summary`
       2. 传递给 Hub 的永远是：`最新变更需求` + `当前代码/文档状态` + `历史精简摘要`
    3. 确定性路由（Deterministic Routing）：不要把所有事情都交给大模型来做选择。



###  3. 对等讨论模式 (Peer-to-Peer)
Agent 之间根据需要自由对话，甚至可以发起讨论和辩论



### 4. 子图嵌套 (Sub-graphs)

- **什么是子图嵌套**：把一组紧密相关的 Agent 聚在一起，打包成一个“子图”。对外，就像是一个单一的 Agent；对内，其实是一个带有专属 Supervisor 的小团队
- **状态隔离（State Isolation）**：每个子图拥有自己独立的 `State`（状态字典）。子图内部打工人天天怎么吵架、生成了多少临时垃圾变量，外部的“大群”根本不需要知道。只有当子图任务彻底完成后，才把最终结果更新到全局的 `State` 中
